# gcsim integration

## Fixed upstream contract

- Repository: <https://github.com/genshinsim/gcsim>
- Version: `v2.43.4`
- Commit: `24042de8ba3243693e97cd7efe22292762b08331`
- Config schema contract: `unversioned-v2.43.4` (upstream has no independent Config schema version, so the pinned tag is the contract)
- JSON result schema: `4.2`
- License: MIT, see `THIRD_PARTY_NOTICES.md`
- CLI: `gcsim -c <server temp config> -out <server temp result>`

`latest`, `main`, self-update (`-update`) and a client-selected binary/path are prohibited. The application does not ship a binary in this repository. Before enabling it, place the official release asset below `vendor/gcsim/v2.43.4/`. The runner selects the platform filename itself and verifies SHA-256 before the first execution.

## Security boundary

The API accepts a strict normalized combat DTO. It rejects unknown root/build fields, so `config`, `command`, `path`, `cookie`, and `uid` cannot enter the generator. IDs are mapped through `GcsimCharacterMapper`, `GcsimWeaponMapper`, and `GcsimArtifactMapper`; unsupported IDs are excluded from simulation and remain eligible for observed/rule recommendations.

The runner uses `spawn(binary, args, { shell: false })`, an application-created temporary directory, fixed filenames, cleanup in `finally`, a stripped environment (`NODE_ENV`, `LANG`, `LC_ALL` only), one worker per process, timeout, config/result/stdout/stderr limits and a semaphore. It never forwards `DATABASE_URL`, HoYoLAB values or the Config body to logs. Safe log fields are event, jobId, attackerId, duration, candidate count and status.

The current worker is process-local. Concurrent enqueue for the same request shares one Promise, and the process admits at most eight active Jobs by default. The interface permits a dedicated container/queue later. Before a multi-instance production rollout, add a durable queue/lease and sandboxed worker rather than relying on an HTTP process remaining alive.

## Config and output

The generator includes character level/ascension cap, constellation, talents, weapon level/ascension/refinement, available artifact set IDs, available artifact stats, level-100 enemies, target count, duration, energy event, iteration count and a trusted rotation template. It never derives a set ID from a localized name.

v2.43.4 does not expose a main-simulation fixed-seed option in its CLI/config parser. The output `sample_seed` is generated by gcsim. This implementation therefore does not claim deterministic RNG; it fixes version, Config and iterations, and records the normalized aggregate. Adding a non-existent seed option would make the Config invalid.

The parser requires result schema `4.2` and the pinned `sim_version` commit, then reads only `statistics.dps.mean`, `iterations`, source reaction means and ending energy means from the bounded JSON result. Full output and Config are not persisted. Cache keys include the tag, commit, Config/result contracts, duration, iterations, normalized build, enemy settings and rotation version.

## Adding support

1. Confirm the numeric ID and key in the pinned official tag's `data_gen.textproto` or artifact `config.yml`.
2. Add the mapping and tests. Never map by Japanese/English display name.
3. Add a trusted rotation at team, attacker or reaction scope. Generic rotation stays low confidence.
4. Run Config Golden, parser, fake runner and security tests.
5. Upgrading gcsim requires a separate review of license, tag commit, checksums, Config parser and result proto.

## Operations

Keep `GCSIM_ENABLED=false` until migration, binary verification and staging fallback tests pass. Defaults: 20 candidates, concurrency 2, eight process-local active Jobs, timeout 30 seconds, 1,000 iterations, 24-hour cache/Job TTL. Expired simulation caches remain available for one additional cache TTL as stale fallback, then are deleted during enqueue. See `.env.example`; invalid ranges fall back to safe defaults. Do not put a binary path in an environment variable.
